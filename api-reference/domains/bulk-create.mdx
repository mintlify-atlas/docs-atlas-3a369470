---
title: "Bulk create domain mappings"
description: "Create multiple domain mappings in a single request"
---

Creates multiple domain-to-backend mappings in one API call. All entries are inserted in a single SQLite transaction — if any entry fails, none are created.

<Note>
  The bulk operation is all-or-nothing. A single invalid or duplicate entry causes the entire request to fail and no mappings are written to the database.
</Note>

## Authentication

All requests require an `Authorization: Bearer <PROXY_API_KEY>` header and must originate from the `PROXY_API_DOMAIN` host. See [Authentication](/api-reference/authentication) for details.

## Request

**POST** `/api/config/bulk`

The endpoint accepts two request body formats.

<Tip>
  Use the array format — it is shorter and requires less nesting.
</Tip>

### Format 1: Array of domain objects (preferred)

Send a JSON array directly in the request body:

```json
[
  {"domain": "example.com", "ip": "192.168.1.100", "port": 8080, "protocol": "http"},
  {"domain": "api.example.com", "ip": "192.168.1.101", "port": 3000}
]
```

### Format 2: Object with `domains` field

Wrap the array in an object with a `domains` key:

```json
{
  "domains": [
    {"domain": "example.com", "ip": "192.168.1.100", "port": 8080, "protocol": "http"},
    {"domain": "api.example.com", "ip": "192.168.1.101", "port": 3000}
  ]
}
```

Both formats are equivalent. The endpoint tries the array format first and falls back to the object format automatically.

### Fields per entry

<ParamField body="domain" type="string" required>
  The domain name to route. Must be unique across all existing mappings.
</ParamField>

<ParamField body="ip" type="string" required>
  The backend IP address to forward traffic to.
</ParamField>

<ParamField body="port" type="number" required>
  The backend port number.
</ParamField>

<ParamField body="protocol" type="string" default="http">
  The protocol used to reach the backend. Accepted values: `http`, `https`. Defaults to `http` if omitted.
</ParamField>

## Response

Returns `201 Created` with an array of the newly created domain objects.

<ResponseField name="domain" type="string" required>
  The domain name for this mapping.
</ResponseField>

<ResponseField name="ip" type="string" required>
  The backend IP address.
</ResponseField>

<ResponseField name="port" type="number" required>
  The backend port number.
</ResponseField>

<ResponseField name="protocol" type="string" required>
  The protocol used to reach the backend.
</ResponseField>

<ResponseField name="created_at" type="string" required>
  ISO 8601 timestamp of when the mapping was created.
</ResponseField>

<ResponseField name="updated_at" type="string" required>
  ISO 8601 timestamp of the most recent update. Equal to `created_at` on creation.
</ResponseField>

## Example

```bash
curl -X POST http://api.example.com/api/config/bulk \
  -H "Authorization: Bearer your-secret-api-key" \
  -H "Content-Type: application/json" \

    {"domain":"site1.example.com","ip":"10.0.0.1","port":8080},
    {"domain":"site2.example.com","ip":"10.0.0.2","port":8081},
```

## Error responses

| Status | Cause |
|--------|-------|
| `400 Bad Request` | Request body is not valid JSON in either accepted format. |
| `400 Bad Request` | The array is empty. |
| `400 Bad Request` | One or more entries are missing `domain`, `ip`, or `port`. The error message includes the zero-based index of the first invalid entry: `Missing required fields in domain at index N`. |
| `500 Internal Server Error` | Database error — for example, a domain in the batch already exists. The entire transaction is rolled back. |
